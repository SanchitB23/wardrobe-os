# Wardrobe OS v2.1.0 — Wear Logs, Acquisitions Pipeline & Observability

**Release date:** 2026-07-12  
**Tag:** `v2.1.0`  
**Previous:** [v2.0.1](./v2.0.1.md)

## Summary

Minor release bundling the post–Inventory Image Intelligence wave: **ad-hoc wear
logs**, the **acquisition → inventory pipeline**, **category optimization**,
and **production observability** — still with deterministic engines deciding and
AI only explaining ([ADR-005](../adr/ADR-005-ai-does-not-decide.md)).

## Product surfaces

- **Wear logs** — `/wear-logs`, `/wear-logs/new`, `/wear-logs/[id]` (+ promote to outfit)
- **Acquisitions pipeline** — Decision History polish; convert wizard; wishlist highlight deep links
- **Category Optimize** — `/intelligence/optimize`
- **Developer** — observability, execution graph, feature flags, runtime statistics, request replay, wear-logs debug

## Architecture Changes

| Layer | Change |
| --- | --- |
| Domain | `src/domain/wear-logs/`; AcquisitionPipeline lifecycle enrichment for UI |
| Persistence | `wear_events` / `wear_event_items`; wishlist purchase/inventory FKs (018C) |
| Service | wear-events orchestration; pipeline CTAs without silent purchase |
| Runtime | AIRuntime metrics/fallback; weather + proxy structured logs; replay store |
| UI | Decision History polish; wear log flows; developer hub pages |

## Migration Notes

1. Apply the two SQL migrations above if missing.
2. Deploy app with `package.json` / tag `v2.1.0`.
3. Optional: set `REPLAY_CAPTURE` only in non-production for request replay.
4. Wear events dual-write to legacy `wear_logs` so analytics engines stay continuous.
